# Документация: интеграция с API PromElec (aaa.na4u.ru / promelec.ru)

## 1. Общие сведения

Интеграция с поставщиком **PromElec** (promelec.ru) реализована через API **aaa.na4u.ru**. Все скрипты обмена расположены в каталоге:

```
/var/www/www-root/data/www/lvtgroup.ru/api/
```

**Используемые URL API:**
- **Продакшен:** `https://aaa.na4u.ru/rpc/`
- **Тестовый:** `https://aaa.na4u.ru/rpc_test/`

**Учётные данные (зашиты в скриптах):**
- Логин: `lvtgroup2`
- Пароль: передаётся в виде **MD5 в верхнем регистре**: `strtoupper(md5('30316'))`
- Идентификатор клиента (customer_id): `148949`

---

## 2. Методы API

По коду используются **два метода** RPC.

### 2.1. `items_data_get` — список товаров с пагинацией

**Назначение:** получение порции товаров для полного импорта или обновления цен/остатков.

**Параметры запроса (JSON в теле POST):**

| Параметр      | Тип    | Обязательный | Описание |
|---------------|--------|--------------|----------|
| `login`       | string | да           | Логин |
| `password`    | string | да           | MD5 пароля (верхний регистр) |
| `method`      | string | да           | `items_data_get` |
| `customer_id` | int    | да           | Идентификатор клиента (148949) |
| `item_id`     | int    | нет          | ID последнего полученного товара — для следующей порции. В первом запросе не передаётся |
| `in_stock`    | string | нет          | `'true'` — только товары в наличии; не передавать или `false` — все товары |

**Пагинация:** ответ содержит массив товаров. Для следующей порции в следующий запрос передаётся `item_id` последнего элемента из текущего ответа. Цикл продолжается, пока API не вернёт пустой массив или не будет передан последний `item_id`.

**Пример первого запроса:**
```json
{
  "login": "lvtgroup2",
  "password": "E3E36F5C779D8A1F738264FD2B8E3222",
  "method": "items_data_get",
  "customer_id": 148949
}
```

**Пример следующего запроса:**
```json
{
  "login": "lvtgroup2",
  "password": "E3E36F5C779D8A1F738264FD2B8E3222",
  "method": "items_data_get",
  "customer_id": 148949,
  "item_id": 9966
}
```

**Ответ:** массив объектов товаров (см. раздел «Структура данных товара»).

---

### 2.2. `item_data_get` — один товар по ID

**Назначение:** отладка и получение данных по одному товару.

**Параметры запроса:**

| Параметр      | Тип    | Обязательный | Описание |
|---------------|--------|--------------|----------|
| `login`       | string | да           | Логин |
| `password`    | string | да           | MD5 пароля (верхний регистр) |
| `method`      | string | да           | `item_data_get` |
| `item_id`     | int    | да           | ID товара в системе поставщика |

**Ответ:** один объект товара (та же структура, что и элемент массива в `items_data_get`).

**Использование в проекте:** `tovar.php`, `debug_api.php` (тест одного товара, например item_id = 3773, 438631).

## 3. Количество и характер запросов

### 3.1. Полный импорт товаров (`load.php`)

- **Метод:** только `items_data_get`.
- **Количество запросов:** не фиксировано. Запросы выполняются в цикле с пагинацией:
  - 1-й запрос — без `item_id`;
  - каждый следующий — с `item_id` последнего товара из предыдущего ответа;
  - цикл завершается, когда получен пустой массив или нет `item_id` для следующей итерации.
- **Лимит:** не более 1000 пакетов за один запуск (защита в коде).
- **Пауза между запросами:** 2 секунды.
- **Таймаут одного запроса:** 600 с (connect 120 с).

Итого: **число запросов = число пакетов до конца выдачи** (зависит от объёма каталога у поставщика).

### 3.2. Обновление цен и остатков (`update_price.php`)

- **Метод:** только `items_data_get` (тот же цикл с пагинацией).
- **Количество запросов:** такое же, как при полном импорте — пока API отдаёт порции товаров.
- **Пауза между запросами:** 2 секунды.
- **Таймаут:** 300 с (connect 60 с).

### 3.3. Вспомогательные скрипты

- **tovar.php** — 1 запрос `item_data_get` с заданным `item_id`.
- **vsetovary.php** — 1 запрос `items_data_get` без `item_id` (первая порция).
- **debug_api.php** — 1 запрос `item_data_get` для товара 3773.

## 4. Куда и как загружаются данные

### 4.1. Полный импорт (`load.php`)

Данные загружаются в:

1. **Инфоблок каталога (IBLOCK_ID = 11)**  
   Создаются/обновляются элементы (товары), разделы (категории), свойства элементов.

2. **Инфоблок брендов (IBLOCK_ID = 6)**  
   При отсутствии бренда с таким именем создаётся новый элемент; свойство товара «Бренд» ссылается на этот элемент.

3. **Модуль каталога Bitrix**  
   - Цены: тип цены ID = 1 (руб.).  
   - Остатки и склады: общее количество и остатки по складам 1 и 2.

4. **Файлы**  
   - Изображение: скачивается по `photo_url`, сохраняется во временный файл, затем через `CFile::MakeFileArray` — в поле `DETAIL_PICTURE` элемента.

**Запуск:** вручную или по cron через скрипт:
```bash
/var/www/www-root/data/www/lvtgroup.ru/api/start_import.sh
```
Скрипт очищает логи и `import_stats.json`, затем вызывает:
```bash
php /var/www/www-root/data/www/lvtgroup.ru/api/load.php
```

### 4.2. Обновление цен и остатков (`update_price.php`)

Обновляются **только существующие** товары (по свойству 501 = внешний ID API):

- Цены (удаление старых цен типа 1, добавление новых из `pricebreaks`).
- Остатки (общее количество и по складам 1 и 2).
- Свойства: бренд (66), дата окончания резерва (1229).
- Поле элемента: `TIMESTAMP_X`.

Новые товары этим скриптом **не создаются**.

**Запуск:**
```bash
/var/www/www-root/data/www/lvtgroup.ru/api/start_price_update.sh
```
Внутри вызывается:
```bash
php /var/www/www-root/data/www/lvtgroup.ru/api/update_price.php
```

## 5. Маппинг: API → инфоблок и каталог Bitrix

### 5.1. Инфоблок товаров (IBLOCK_ID = 11)

| Поле/свойство элемента Bitrix | ID свойства | Поле API / источник | Примечание |
|-------------------------------|-------------|----------------------|------------|
| NAME                          | —           | `name` + при наличии `description`: `"name - description"` | Строка |
| CODE                          | —           | Транслит от `name`   | Автогенерация |
| XML_ID                        | —           | `'API_' . item_id`   | Уникальный внешний идентификатор |
| ACTIVE                        | —           | `'Y'`                | Все активны |
| PREVIEW_TEXT / DETAIL_TEXT    | —           | `description`        | Описание |
| IBLOCK_SECTION                | —           | `class0name`, `class1name`, `class2name` | Иерархия из 3 уровней; разделы создаются при отсутствии |
| DETAIL_PICTURE                | —           | `photo_url`          | Скачивание по URL, сохранение через CFile |
| **Внешний ID**                | **501**     | `item_id`            | Идентификатор товара в API (поиск при обновлении) |
| **Название (дубль)**          | **627**     | `name`               | Короткое название |
| Бренд                         | **66**      | `producer_name`      | Ссылка на элемент инфоблока брендов (6); при отсутствии бренд создаётся |
| Замечание / тип               | **494**     | `remark`             | Список (enum), при отсутствии значения создаётся |
| Год выпуска                   | **495**     | `year_of_issue`      | Список (enum) |
| Упаковка (тип)                | **496**     | `package`            | Список (enum) |
| Количество в упаковке         | **497**     | `pack_quant`         | Список (enum) |
| Тип упаковки                  | **498**     | `packaging`          | Список (enum) |
| Вес                           | **499**     | `weight`             | Список (enum) |
| Документы (ссылки)            | **1219**    | `datasheet`          | Множественное; ссылка на техдокументацию |
| Дата окончания резерва (EDGE) | **1229**    | `edge`               | Дата окончания резерва |

Категории (разделы) строятся так:
- 1-й уровень: `class0name` (при пустом — «Без категории»).
- 2-й уровень: `class1name`, родитель — раздел 1-го уровня.
- 3-й уровень: `class2name`, родитель — раздел 2-го уровня.  
Товар привязывается к разделу 3-го уровня.

### 5.2. Цены (модуль catalog)

- **Тип цены:** ID = 1, валюта RUB.
- **Источник:** массив `pricebreaks` из API.
- Структура элемента: `quant` — количество от; следующая запись задаёт диапазон до (quant - 1).
- Для каждой пары «количество от — цена» создаётся запись в `b_catalog_price` с полями:
  - `QUANTITY_FROM`, `QUANTITY_TO`
  - `PRICE`, `CURRENCY` = 'RUB'

При обновлении товара старые цены типа 1 удаляются, затем добавляются новые из текущего ответа API.

### 5.3. Остатки и склады

- **Общее количество (QUANTITY):** сумма по всем `vendors`, где `vendor != 0` (т.е. не «резерв»).
- **Склад ID = 1:** сумма по `vendors` с `vendor == 0` (условно «резерв»).
- **Склад ID = 2:** то же, что общее количество (сумма по vendor != 0).

Структура элемента `vendors`: массив с полями `vendor`, `quant`.

### 5.4. Инфоблок брендов (IBLOCK_ID = 6)

- По полю API `producer_name` ищется элемент с таким NAME в инфоблоке 6.
- Если не найден — создаётся новый элемент (NAME, CODE по транслиту), его ID подставляется в свойство 66 товара.

| Файл | Назначение |
|------|------------|
| `load.php` | Полный импорт: товары, разделы, свойства, цены, остатки, картинки, бренды. Один цикл пагинации по `items_data_get`. |
| `update_price.php` | Обновление цен, остатков, бренда и даты резерва у уже существующих товаров (по свойству 501). Тот же цикл `items_data_get`. |
| `tovar.php` | Тест: один запрос `item_data_get` для заданного `item_id` (в коде 438631), вывод JSON. Использует `rpc_test/`. |
| `vsetovary.php` | Тест: один запрос `items_data_get` без пагинации, вывод первых 5 товаров. Использует `rpc_test/`. |
| `debug_api.php` | Отладка: один запрос `item_data_get` для item_id 3773, проверка полей (producer_name, edge и др.). Использует `rpc/`. |
| `start_import.sh` | Запуск полного импорта: очистка логов и статистики, вызов `php load.php`. |

## 7. Краткая схема работы скриптов

### 7.1. load.php (полный импорт)

1. Подключение Bitrix (iblock, catalog), инициализация статистики и логов.
2. Цикл запросов `items_data_get` с пагинацией (`item_id` после первого запроса).
3. Для каждой порции:
   - По массиву `item_id` — пакетный поиск существующих товаров по свойству 501.
   - Предзагрузка брендов по `producer_name` из инфоблока 6.
   - Для каждого товара:
     - Получение/создание разделов по class0/1/2 name.
     - Подготовка полей и свойств (501, 627, 494–499, 1219, 1229, бренд 66), картинка по `photo_url`.
     - Создание или обновление элемента (Update по существующему ID или Add).
     - Установка бренда (66) и даты резерва (1229).
     - Обновление документов (1219) по `datasheet`.
     - Удаление старых цен типа 1, добавление новых из `pricebreaks`.
     - Обновление остатков и складов по `vendors`.
4. Пауза 2 сек, переход к следующей порции до пустого ответа или лимита 1000 пакетов.
5. Сохранение итоговой статистики в `import_stats.json` и вывод итогов в логи.

### 7.2. update_price.php (обновление цен и остатков)

1. Подключение Bitrix, инициализация логов и статистики.
2. Тот же цикл `items_data_get` с пагинацией.
3. Для каждой порции:
   - Поиск существующих товаров по свойству 501 (только уже созданные в каталоге).
   - Предзагрузка брендов.
   - Для каждого найденного товара:
     - Обновление цен (удаление старых по типу 1, добавление из `pricebreaks`).
     - Обновление остатков и складов по `vendors`.
     - Обновление TIMESTAMP_X.
     - Обновление бренда (66) и даты резерва (1229).
4. Пауза 2 сек между пакетами, сохранение статистики в `price_update_stats.json`.

## 8. Технические параметры запросов (cURL)

- **Content-Type:** `application/json`
- **Метод:** POST
- **Тело:** JSON с полями login, password, method, customer_id и при необходимости item_id, in_stock
- **load.php:** CURLOPT_TIMEOUT = 600, CURLOPT_CONNECTTIMEOUT = 120
- **update_price.php:** CURLOPT_TIMEOUT = 300, CURLOPT_CONNECTTIMEOUT = 60
- **SSL:** CURLOPT_SSL_VERIFYPEER = false, CURLOPT_SSL_VERIFYHOST = false
