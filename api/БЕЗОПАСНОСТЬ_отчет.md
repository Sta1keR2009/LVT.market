## 🔴 КРИТИЧЕСКИЕ УЯЗВИМОСТИ

### 1. Открытый доступ к скриптам импорта и обновления

**Файлы:**
- `load.php` — полный импорт товаров
- `update_price.php` — обновление цен и остатков

**Проблема:** Скрипты можно вызвать через HTTP без авторизации. Любой может:
- Запустить полный импорт (нагрузка на сервер, БД)
- Изменить цены и остатки всех товаров
- Вызвать DoS-атаку на сервер

**Пример атаки:**
```bash
curl https://lvtgroup.ru/api/load.php
curl https://lvtgroup.ru/api/update_price.php
```

**Рекомендация:** Добавить проверку на CLI-режим или авторизацию:
```php
// В начале файла
if (php_sapi_name() !== 'cli') {
    // Проверка авторизации или IP-адреса
    die('Доступ запрещен');
}
```

### 4. Уязвимость в parser/index.php

**Файл:** `api/parser/index.php`

**Проблемы:**

**4.1. Небезопасная десериализация:**
```php
$query = base64_decode($_POST['query']);
$_POST = unserialize($query, ['allowed_classes' => false]);
```
- Можно передать вредоносную сериализованную строку
- Риск выполнения произвольного кода (Object Injection)

**4.2. Brute-force атака на авторизацию:**
```php
$arAuthResult = $USER->Login($_POST['login'], $_POST['password'], "Y");
```
- Нет ограничения попыток входа
- Нет капчи
- Можно подбирать пароли

**4.3. Открытый доступ к данным инфоблоков:**
- Эндпоинт `?get=iblocks` возвращает список всех инфоблоков
- Эндпоинт `?get=data` возвращает разделы, свойства, цены
- Доступно без авторизации после успешного входа

**Рекомендация:**
- Добавить rate limiting для авторизации
- Убрать десериализацию или использовать безопасный формат (JSON)
- Добавить проверку прав доступа для каждого эндпоинта

### 5. Хардкодные учетные данные

**Проблема:** Логин и пароль API PromElec хранятся в открытом виде в коде:

**Файлы с паролями:**
- `load.php`: `$login = 'lvtgroup2'; $password = '30316';`
- `update_price.php`: `$login = 'lvtgroup2'; $password = '30316';`
- `tovar.php`: `$login = "lvtgroup2"; $password = "30316";`
- `vsetovary.php`: `$login = "lvtgroup2"; $password = "30316";`
- `debug_api.php`: `$login = 'lvtgroup2'; $password = '30316';`

**Риски:**

- Вынести учетные данные в конфигурационный файл вне веб-доступа:

  ```php
      'login' => 'lvtgroup2',
      'password' => '30316',
      'customer_id' => 148949
  ];
  ```
- Или использовать переменные окружения
- Добавить файл в `.gitignore`

### 6. Отсутствие .htaccess для ограничения доступа

**Проблема:** В директории `/api/` нет файла `.htaccess` для ограничения доступа.

**Рекомендация:** Создать `.htaccess` с правилами:
```apache
# Запретить доступ ко всем PHP файлам кроме разрешенных
<FilesMatch "\.php$">
    Order Deny,Allow
    Deny from all
    # Разрешить только для CLI или определенных IP
    Allow from 127.0.0.1
    Allow from ::1
</FilesMatch>

# Или требовать авторизацию
<FilesMatch "\.php$">
    AuthType Basic
    AuthName "API Access"
    AuthUserFile /path/to/.htpasswd
    Require valid-user
</FilesMatch>
```

## 🔒 РЕКОМЕНДУЕМЫЕ ИСПРАВЛЕНИЯ (примеры кода)

### Пример 1: Защита скриптов импорта (load.php, update_price.php)

```php
<?php
// В начале файла, после установки DOCUMENT_ROOT

// Проверка на CLI-режим
if (php_sapi_name() !== 'cli') {
    // Альтернатива: проверка IP
    $allowedIPs = ['127.0.0.1', '::1', '10.0.0.0/8']; // внутренние IP
    $clientIP = $_SERVER['REMOTE_ADDR'] ?? '';
    $allowed = false;
    
    foreach ($allowedIPs as $ip) {
        if (strpos($ip, '/') !== false) {
            // CIDR проверка
            if (ip_in_range($clientIP, $ip)) {
                $allowed = true;
                break;
            }
        } else {
            if ($clientIP === $ip) {
                $allowed = true;
                break;
            }
        }
    }
    
    if (!$allowed) {
        http_response_code(403);
        die('Доступ запрещен. Скрипт доступен только из CLI или с разрешенных IP.');
    }
}

// Остальной код...
```
